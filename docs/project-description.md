# Issue Lens 项目描述与面试讲稿

## 一句话介绍

Issue Lens 是一个证据优先、默认只读的 GitHub Issue 管理 Agent：它结合 GitHub 数据、仓库代码检索和 DeepSeek，生成可审计的 Issue 分析报告与回复草稿，并通过人工审批门禁控制真实评论发布。

## 简历项目描述（推荐版）

**Issue Lens｜GitHub Issue 管理 Agent**  
技术栈：Python、FastAPI、DeepSeek API、GitHub REST API、SQLite、Docker、GitHub Actions

- 设计并实现端到端 Issue 分诊工作流，支持类型/优先级/模块分类、重复 Issue 检测、相关代码定位、复现信息整理、修复建议和回复草稿生成，并支持最多 10 条 Open Issues 顺序分析。
- 构建“规则召回 + LLM 结构化增强 + 强校验回退”链路：先从 Issue 历史和仓库快照召回证据，再由 DeepSeek 生成结构化结果；异常响应自动重试并回退到确定性分析，降低幻觉和外部 API 故障影响。
- 实现人工审批与最小权限写入机制：分离 GitHub 读写客户端和 Token，真实评论需通过草稿校验码、显式模式切换及目标短语三重确认；使用 SQLite 保存审批状态和审计事件。
- 使用 FastAPI 搭建本地 Web 工作台，配套 Docker Compose 和 GitHub Actions；35 项自动化测试覆盖分析、模型校验、GitHub 权限边界、审批状态机和 Web API，并设置离线评测质量门槛。

## 简历项目描述（精简版）

开发 GitHub Issue 管理 Agent，结合 GitHub REST API、仓库代码检索和 DeepSeek，实现 Issue 分类、重复检测、相关文件定位、复现/修复建议与回复草稿生成。采用结构化输出校验、规则回退、读写 Token 分离、人工二次确认和 SQLite 审计机制控制模型幻觉与写入风险，并通过 Docker、GitHub Actions 和 35 项自动化测试保障交付质量。

## 招聘网站项目介绍

独立完成一个面向开发团队的 GitHub Issue 分析与审批工具。系统可以读取公开仓库的 Issue、评论和代码快照，自动完成分诊、重复检测、代码定位、复现信息整理、修复方向生成及回复草拟。分析过程强调证据可追溯：模型只能基于召回的 Issue 与代码上下文生成结构化结果，校验失败时自动回退到规则分析。

项目同时实现了完整的人机协作闭环。用户可以在 Web 工作台中批量选择 Issue、查看分析证据、编辑回复并进行模拟审批；真实 GitHub 评论默认关闭，启用后仍需独立写入 Token、草稿校验码和目标级确认短语。审批记录保存在 SQLite 中，便于追踪每次分析和提交状态。项目提供 Docker 本地运行方式，并使用 GitHub Actions 自动执行测试、离线质量评测和镜像构建。

## 90 秒面试介绍

我做的项目叫 Issue Lens，是一个 GitHub Issue 管理 Agent。它解决的问题是：维护者面对 Issue 时，需要在历史讨论和代码仓库之间反复查找信息，而且直接让大模型自动回复又存在幻觉和误操作风险。

我的处理方式不是把整件事完全交给模型，而是先建立确定性的证据链。系统通过 GitHub API 获取 Issue、评论和历史候选，再浅克隆公开仓库，检索可能相关的源码、测试和依赖文件。在这个基础上，DeepSeek 只负责生成规定结构的分类、复现信息、修复建议和回复草稿。如果输出为空、JSON 不合法或包含没有证据支持的结论，系统会重试并回退到规则分析。

写入侧我做了单独的安全边界。读取和写入使用不同客户端与 Token，默认只允许模拟提交；真实评论必须由用户审核草稿、确认校验码、切换发布模式，并输入包含仓库和 Issue 编号的确认短语。系统目前有 Web 工作台、SQLite 审计记录、Docker 环境、GitHub Actions，以及 35 项自动化测试。这个项目主要体现了我对 Agent 工作流、检索增强、可靠性设计和工程交付的理解。

## 项目难点与回答思路

### 1. 为什么不直接把整个仓库交给大模型？

完整仓库通常超过上下文限制，而且会增加成本和噪声。项目先通过关键词、文件类型和 Issue 语义召回少量候选文件与匹配行，再把证据片段交给模型，提高相关性并保留可解释依据。

### 2. 如何降低模型幻觉？

使用受约束的结构化输出、字段校验、证据一致性检查和维护者身份校验。无依据的版本承诺或项目历史结论会被拒绝；模型连续失败时回退到规则分析，而不是返回未经验证的内容。

### 3. 为什么需要人工审批？

Issue 回复属于对外操作，可能影响项目协作和维护者信誉。模型适合生成建议，但最终发布应由人负责。因此系统默认只读和模拟提交，并对真实写入设置不可跳过的确认链路。

### 4. 为什么分离读取与写入 Token？

这是最小权限原则。日常分析只需要读取权限；只有明确启用发布时才加载单独的写入 Token。即使读取链路或模型处理出现问题，也不能直接获得 GitHub 写权限。

### 5. 当前项目还有哪些不足？

目前只支持公开仓库快照，批量分析仍由浏览器顺序执行，评测集规模也较小。后续可以加入私有仓库认证、后台任务队列、更完整的检索索引，以及基于真实失败案例持续扩充的评测体系。
